# WP-06B2 Canonical SessionRepository Manual Tests

## Preconditions

- Build with `rojo build default.project.json --output /tmp/LCA-WP06B2.rbxlx`.
- Test in an isolated Studio place with real test Players and cloned ModuleScripts where fresh private state is required.
- Use the WP-06B1 ServerDataService with in-memory persistence and synchronization callbacks for integration cases.

## Module and API Isolation

- [ ] Requiring the module only requires Config and UpgradeDefinitions and creates no gameplay or external runtime effects.
- [ ] The frozen module exports exactly `createSession`, `getSession`, `removeSession`, `getDefaultData`, `migrateData`, `buildSyncPacket`, and `buildQuestSyncPacket`.
- [ ] No sessions map, `getAllSessions`, raw-data accessor, dirty helper, state setter, or test reset is exported.
- [ ] Runtime source has no DataStore API, lifecycle connection, task, RemoteEvent discovery, Workspace access, UI, reward, purchase, receipt, or quest gameplay logic.

## Player Validation and Session Ownership

- [ ] Every Player-taking API rejects nil, numbers/userIds, strings, tables, and non-Player Instances.
- [ ] `createSession` returns one session for a valid Player and stores it by numeric UserId.
- [ ] Initial `data` is a unique empty table.
- [ ] Initial metadata is exactly Loading, revision 0, savedRevision 0, dirty false, saveInFlight false, finalizeRequested false, and lastResult nil.
- [ ] A duplicate call for the same Player returns nil and does not overwrite the first wrapper.
- [ ] A different Player object with the same UserId cannot create, get, or remove the first Player's session.
- [ ] A stored Released wrapper is not replaced directly; B1 must remove it before creating its replacement.
- [ ] `getSession` returns the live wrapper only to the exact Player; document the fixture as trusted server orchestration.
- [ ] `removeSession` returns true only once for the exact Player and false for absent, invalid, or identity-mismatched calls.
- [ ] Dirty remains a compatibility mirror of `revision > savedRevision`; repository methods never author revision transitions.

## Fresh Complete Defaults

- [ ] Every default contains the exact recovered top-level fields and no calculated ClickPower, AutoPower, CoreAmplifier, or Luck fields.
- [ ] UpgradeLevels has exactly the four required known IDs with zero defaults.
- [ ] Reward, playtime, factory, quest, achievement, collection, daily-login, lifetime-counter, and eight-slot RarityCount defaults match the specification.
- [ ] Two calls share no top-level or nested tables; mutation of one default cannot affect another.
- [ ] FirstJoin is a finite non-negative integer generated by `getDefaultData`.
- [ ] DataVersion equals Config.DataVersion.

## Migration and Change Detection

- [ ] Migrating an untouched fresh canonical default returns owned data, `changed=false`, and nil error.
- [ ] Cloning alone does not set changed and output mutation cannot affect input.
- [ ] Removing each known field individually adds its default and returns changed true, except FirstJoin which fails closed.
- [ ] Invalid finite/non-finite known numbers normalize to deterministic defaults or approved Config caps and return changed true.
- [ ] Fractions floor, negatives clamp to zero/minimum, and numeric strings do not become authority.
- [ ] Wrong-type required nested structures are repaired with fresh tables and return changed true.
- [ ] Missing nested known fields, all four upgrade IDs, and RarityCount slots 1..8 are repaired and return changed true.
- [ ] Valid current-version canonical input remains changed false.
- [ ] Missing/older DataVersion routes as legacy, advances only after successful normalization, and returns changed true.
- [ ] Invalid DataVersion returns `INVALID_DATA_VERSION`.
- [ ] DataVersion above Config.DataVersion returns `FUTURE_DATA_VERSION` without producing data.
- [ ] Existing missing FirstJoin returns `MISSING_FIRST_JOIN` and does not fabricate a timestamp.
- [ ] Existing invalid FirstJoin returns `INVALID_FIRST_JOIN` and does not fabricate a timestamp.
- [ ] A valid FirstJoin is preserved exactly.
- [ ] Caller input and all descendants remain byte-for-byte/logically unchanged after both successful and failed migration.
- [ ] Safe unknown top-level and nested fields survive unchanged and do not alone set changed.
- [ ] Safe legacy CriticalChance and CriticalMultiplier fields are preserved.
- [ ] Cycles, NaN, both infinities, unsupported values, unsupported keys, and non-integer numeric keys fail closed.
- [ ] Depth at the documented boundary succeeds and deeper data returns `MAX_DEPTH_EXCEEDED`.

## Main Sync Packet

- [ ] Invalid Player, absent session, Loading, LoadFailed, Released, or inconsistent dirty metadata returns nil.
- [ ] Loaded and Saving sessions can build packets without mutating data or metadata.
- [ ] Packet fields are exactly Energy, LifetimeEnergy, four calculated stats, Rebirths, Gems, UpgradeLevels, DailyReward, PlaytimeReward, PurchasedPerks, History, BestRarity, BestRarityName, FactoryStage, and HighestFactoryStage.
- [ ] Calculated stats equal `UpgradeDefinitions.calculateStats(UpgradeLevels, Rebirths)` and are not added to session data.
- [ ] No FirstJoin, DataVersion, quest data, unknown top-level field, or session metadata leaks.
- [ ] Unknown fields nested beside known UpgradeLevels/reward/playtime fields do not leak.
- [ ] Mutating any packet descendant cannot mutate the session, another packet, or a later packet.
- [ ] Invalid packet-source values cause nil rather than a partial packet.

## Quest Sync Packet

- [ ] Invalid Player, absent session, wrong state, or inconsistent dirty metadata returns nil.
- [ ] Packet fields are exactly DailyQuests, Achievements, Collection, DailyLogin, and Stats.
- [ ] DailyQuests contains Date, Quests, SessionStart; Achievements contains Unlocked, Claimed; Collection contains four recovered maps; DailyLogin contains three recovered fields.
- [ ] Stats contains the confirmed press, energy, rebirth, factory, five rarity-total, and eight RarityCount fields.
- [ ] No definitions, rewards, unknown top-level field, unknown sibling field, or session metadata leaks.
- [ ] Mutating packet descendants cannot mutate session data or any other packet.
- [ ] Packet building does not grant progress, complete quests, or change any gameplay value.

## WP-06B1 Compatibility and Exclusions

- [ ] Inject the module as B1's `sessions` dependency without aliases or adapter methods.
- [ ] Exercise new, unchanged, changed, failed, duplicate, sync, save, and final-save-retry flows.
- [ ] B1 can replace `session.data` and update the exact required metadata fields.
- [ ] Failed final save leaves the exact session available, Loaded, dirty, and retryable.
- [ ] No DataStore, autosave, retry, cross-server lock, PlayerAdded, PlayerRemoving, BindToClose, Main.server, PlayerDataService, FactoryEvolution, MainGuiClient, remote, or Shared/LCA_Shared bridge behavior exists.
- [ ] Only the WP-06B2 implementation allowlist changed and `git diff --check` plus Rojo build pass.
